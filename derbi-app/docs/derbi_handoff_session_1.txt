### 1. Spline 3D Model: Autonomous Interaction & Raycaster Waking (`experience-hero.jsx`)

**The Goal:** Programmatically animate the 3D model using synthetic pointer events, but strictly pause the automation loop for 5 seconds whenever a human user interacts with the canvas.

**The Fixes Implemented:**
- **Splash Screen Synchronization:** The automated sequence is strictly locked behind a `waitForAppEntry()` promise loop that constantly checks `document.body.style.overflow`. It mathematically guarantees the 3D model does not begin animating until the Splash Screen animation has completely finished and unblocked the DOM.
- **The Timestamp Poller (`Date.now()`):** Ripped out brittle boolean flags (`isUserInteracting`) because Spline's internal event suppressors were causing micro-second race conditions where `pointerleave` fired too early. Built an impenetrable condition: `Date.now() - lastInteractionTime < 5000`.
- **Bounding Box Isolation (`isPointerInsideSpline`):** Ensuring users scrolling text on the left hand side of the screen don't silence the Spline model on the right. The timestamp is ONLY updated if native `pointerenter` and `pointerleave` events confirm the cursor is physically inside the `<spline-viewer>` boundary.
- **Waking the WebGL Raycaster:** Spline puts its physics raycaster to sleep to save memory when a mouse leaves the screen. Synthesizing `pointermove` was not enough. Programmed the automation sequence to dispatch a faux `pointerover` and `pointerenter` exactly on Frame 1 of the animation to "wake up" the engine, followed by a faux `pointerout` and `pointerleave` at the end to put the engine back to sleep.
- **DOM Event Capture Phase:** Bound native interaction listeners (`pointermove`, `wheel`, `touchstart`) strictly to the `viewer` node using `{ capture: true, passive: true }` to guarantee React logs the human interaction nanoseconds before Spline’s shadow DOM consumes/stops the event propagation.

### 2. "Our Offerings" Animated SVG Section (`our-offerings.jsx`)

**The Goal:** Create a 10-grid layout with Pitch Black card hover states, housing locally hosted Lottie animations that turn pure white on hover. 

**The Fixes Implemented:**
- **Local Asset Architecture:** 10 Lottie JSON files were downloaded away from WordPress and stored locally in `public/lottie/` to drastically improve load speeds and prevent external CORS failures. 
- **Legacy After Effects Corruption (The Silent Crash):** `research.json` (Clinical Trials) and `business-network.json` (Networking) were completely crashing the `@lottiefiles/react-lottie-player` into an invisible state without throwing console errors.
  - **Node.js JSON Scrubbing:** Built a Node script that parsed the corrupted 1.5MB JSONs and forcefully stripped out invalid `<image>` tags and legacy `markers` generated by a 2019 version of AfterEffects (v3.1.1) that modern players violently reject.
- **Forced First-Frame Rendering (`autoplay` hack):** Because the legacy AfterEffects metadata was pruned, the modern Lottie player bugged out and refused to paint the first SVG frame to the screen until the mouse physically hovered on it.
  - **Solution:** Configured the `Player` component to hook into its native `onEvent={(event) => event === 'load'}`. The exact millisecond the Lottie fully downloads, the script programmatically triggers `.play()`, calculates exactly 800ms of playback (to clear the corrupted frames and paint the paths to the canvas), and then calls `.stop()`. This operates as a robust fallback to ensure elements are always visible, while doubling beautifully as a subtle entrance animation.
- **CSS Pitch-Black Inversions:** Rather than recoloring 10 massive JSON files, utilized Tailwind's `group-hover:brightness-0 group-hover:invert group-hover:drop-shadow-[0_0_12px_rgba(255,255,255,0.9)]` to instantaneously mathematically flip the SVG strokes from black to glowing white when the Pitch Black hover state activates.
